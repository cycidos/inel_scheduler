# Work Scheduler 앱 계획서 (v1)

> 범위: 협업 스케줄러 단독 앱(Electron)  
> 대상: 이늘(관리자) + 편집자  
> 참고 UI 톤: 레퍼런스 앱(White + Soft Pink)

---

## 1) 제품 방향

- 본 단계는 `협업 스케줄러`를 `방송 스케줄러`와 분리하여 단독 앱으로 개발한다.
- 앱 설치형 배포를 기본으로 하며, 사용자는 설치 후 exe 실행으로 접근한다.
- 데이터 원천은 Google Sheets 링크이며, 앱에서 가져오기/내보내기를 제공한다.
- **배포 단계 분리**:
  - **1차 배포**: 관리자(이늘) 단독 사용. 효율적인 일정 관리가 목표.
  - **2차 배포**: 편집자/썸네일러 협업 가능. 1차 코드 위에 권한 모드/install 빌드/부분 업데이트 추가.

---

## 2) 핵심 요구사항 (확정)

1. 설정에서 Google Sheets 링크를 입력해 기본 데이터 소스로 사용
2. 앱에서 `가져오기(시트 -> 앱)` / `내보내기(앱 -> 시트)` 수행
3. 테이블 헤더(컬럼) 추가/삭제 가능
4. 헤더 추가 시 각 컬럼의 타입 지정 가능 (Notion DB 스타일)
5. 치지직 방송 링크 기반 업타임 타임라인 수집
   - 방송 제목/카테고리 변경 이력 누적
   - 감지 간격 설정: `0.5초 ~ 30초`
   - 영상 녹화 기능은 포함하지 않음
6. 사용자 전달물은 설치 파일(설치 후 exe 실행)
7. UI는 Electron 기반
8. UI 테마는 White + Soft Pink
9. UI 구조는 상단 탭형 전환
   - `숏폼 / 롱폼 / 다시보기`
   - 사이드바 메뉴는 사용하지 않음

## 4) 테이블별 기본 헤더 및 타입 정의

## 4-1. 타입 시스템 정의 (컬럼 타입)

- `text`: 일반 문자열
- `long_text`: 긴 텍스트/메모
- `select`: 단일 선택
- `multi_select`: 다중 선택
- `date`: 날짜
- `datetime`: 날짜+시간
- `status`: 상태값 전용 선택
- `url`: 링크
- `boolean`: 체크박스
- `number`: 숫자
- `timeline`: 업타임 기반 이벤트 누적 — **한 셀 다중행 텍스트** 형식 (`HH:MM:SS - 카테고리`, 줄바꿈 누적)

## 4-2. 숏폼/롱폼 2행 구조 (영상 단위 + 담당자별 행)

### 핵심 개념

하나의 영상(task)은 **2행**으로 구성된다:
- **1행째 (썸네일러)**: 영상 공통 정보 + 썸네일러의 작업 정보
- **2행째 (영상편집자)**: 영상편집자의 작업 정보

`업로드`, `방송날짜`, `영상제목`, `영상 카테고리`는 **영상 단위(1행)**에 속하며,
`담당자` 이후 열은 **담당자별(2행)**로 독립적이다.

### 헤더 정의 (숏폼/롱폼 공통)

| 헤더 | 행 범위 | 타입 | 값/규칙 |
|------|---------|------|---------|
| 업로드 | 영상 단위 (1행) | status | 완 / (빈값) — 유튜브 업로드 완료 시 "완" 표시, 미완이면 비워둠 |
| 방송날짜 | 영상 단위 (1행) | date | YYYY-MM-DD |
| 영상제목 | 영상 단위 (1행) | text | 필수 권장 |
| 영상 카테고리 | 영상 단위 (1행) | select | 게임 / 노래음악 / 토크 (태그 형태, 추가 가능) |
| 담당자 | 담당자별 (2행) | text | 1행: 썸네일러 이름, 2행: 영상편집자 이름 |
| 시작일 | 담당자별 (2행) | date | 작업 시작일 |
| 마감일 | 담당자별 (2행) | date | 작업 마감일 |
| 진행상태 | 담당자별 (2행) | status | Wait / Wip / Done / Retake / Omit |
| 납품상태 | 담당자별 (2행) | status | Wait / Done / Publish |
| 공유링크 | 담당자별 (2행) | url | 원본 공유 링크 |
| 납품링크 | 담당자별 (2행) | url | 납품상태=Publish일 때만 활성 |

### 관리자 앱에서 보이는 형태

```
| 업로드 | 방송날짜   | 영상제목            | 영상 카테고리 | 담당자         | 시작일 | 마감일 | 진행상태 | 납품상태 | 공유링크 | 납품링크 |
|-------|-----------|--------------------|-----------  |---------------|--------|--------|---------|---------|---------|---------|
| 완    | 2026-04-08| 이늘 게임 하이라이트  | 게임         | 썸네일러 김OO   | 04-08  | 04-10  | Done    | Publish | link1   | link2   |
|       |           |                    |             | 영상편집자 박OO | 04-08  | 04-12  | Wip     | Wait    | link3   |         |
|       | 2026-04-09| 이늘 노래 모음       | 노래음악      | 썸네일러 김OO   | 04-09  | 04-11  | Wait    | Wait    |         |         |
|       |           |                    |             | 영상편집자 박OO | 04-09  | 04-13  | Done    | Publish | link5   | link6   |
```

- `업로드` / `방송날짜` / `영상제목` / `영상 카테고리`는 rowspan 병합 (1행)
- `담당자` 이후는 2행으로 분리
- 2행째의 `업로드` / `방송날짜` / `영상제목` / `영상 카테고리`는 비워둠

### 편집자(영상편집자) 앱에서 보이는 형태

자기 역할 행만 필터 → **영상 1개 = 1행**:

```
| 업로드 | 방송날짜   | 영상제목            | 영상 카테고리 | 담당자         | 시작일 | 마감일 | 진행상태 | 납품상태 | 공유링크 | 납품링크 |
|-------|-----------|--------------------|-----------  |---------------|--------|--------|---------|---------|---------|---------|
| 완    | 2026-04-08| 이늘 게임 하이라이트  | 게임         | 영상편집자 박OO | 04-08  | 04-12  | Wip     | Wait    | link3   |         |
|       | 2026-04-09| 이늘 노래 모음       | 노래음악      | 영상편집자 박OO | 04-09  | 04-13  | Done    | Publish | link5   | link6   |
```

### 썸네일러 앱에서 보이는 형태

```
| 업로드 | 방송날짜   | 영상제목            | 영상 카테고리 | 담당자        | 시작일 | 마감일 | 진행상태 | 납품상태 | 공유링크 | 납품링크 |
|-------|-----------|--------------------|-----------  |--------------|--------|--------|---------|---------|---------|---------|
| 완    | 2026-04-08| 이늘 게임 하이라이트  | 게임         | 썸네일러 김OO  | 04-08  | 04-10  | Done    | Publish | link1   | link2   |
|       | 2026-04-09| 이늘 노래 모음       | 노래음악      | 썸네일러 김OO  | 04-09  | 04-11  | Wait    | Wait    |         |         |
```

> `업로드` 상태는 관리자/편집자/썸네일러 모든 앱에서 볼 수 있다 (읽기 전용, 관리자만 변경 가능).

### Google Sheets 저장 구조

시트에서는 물리적으로 **영상 1개 = 2행**:
- 1행: 영상 공통 정보 + 썸네일러 작업 정보
- 2행: `업로드`/`방송날짜`/`영상제목`/`영상 카테고리` 비워둠 + 영상편집자 작업 정보
- 같은 영상의 행 쌍은 `방송날짜`가 있는 행 = 그룹의 시작으로 식별

## 4-3. Full Replay (다시보기) 기본 헤더 타입

| 헤더 | 타입 | 값/규칙 |
|---|---|---|
| 업로드 | status | 완 / (빈값) — 유튜브 업로드 완료 시 "완" 표시 |
| 방송일 | date | YYYY-MM-DD (방송 시작일, 감지 시 자동 기록. 자정을 넘겨도 시작 당일 유지) |
| 방송시작시간 | text | KST(Asia/Seoul) 24시 기준 `HH:MM`. 같은 날 방송이 여러 번이면 행 구분의 보조키 역할 |
| 영상제목 | text | 방송 제목 (감지 시 첫 제목 자동 기록) |
| 카테고리 타임라인 | text(다중행) | 한 셀 다중행 텍스트, `HH:MM:SS - 카테고리` 줄바꿈 누적 |

### 행 분리 규칙 (자동 감지)
- **1행 = 1방송 세션** (LIVE ON → OFF 까지). 자정을 넘겨도 같은 행에 누적.
- LIVE OFF 가 감지되면 행을 닫고, 다음 LIVE ON 에서 새 행을 만든다.
- 같은 날 방송이 여러 번이면 시작시간이 다른 새 행이 추가된다 (휴식 = 분할).
- **단절 시 같은 세션 자동 이어쓰기**: 인터넷 끊김, 앱 재시작, 방송감지 OFF→ON 같은
  외부 단절로 LIVE OFF 가 잠깐 발생하더라도, 치지직 API 의 `openDate`(방송이 켜진 시각)
  는 같은 방송 동안 변하지 않는다. 따라서 다음 LIVE 감지 시 그 `openDate` 로 만든
  핑거프린트 `(broadcastDate, broadcastStartTime[HH:MM:SS])` 가 다시보기 탭의 기존
  행과 매칭되면 새 행을 만들지 않고 **같은 행에 이어서 카테고리 타임라인을 누적**한다.
- **방송시작시간은 초까지(HH:MM:SS)** 저장한다. openDate 가 그대로 immutable 한 키
  역할을 하도록 정밀도를 낮추지 않음.
- **매칭 안전장치**:
  1) 활성 세션(`detectRowId.current`) 동안에는 매칭 비교 자체를 건너뛴다 (매 polling no-op)
     → 같은 세션 polling 중 openDate 가 만에 하나 흔들려도 같은 행 유지.
  2) 매칭 단계: **정확(HH:MM:SS) → 분 단위(HH:MM) prefix → 실패 시 새 행**.
     → 1분 이내 미세 흔들림이나 구 시트 데이터(분 단위)와도 호환.
  3) 운영 검증용으로 같은 세션 내 openDate 변화가 발생하면 디버그 패널에 즉시 경고
     로그를 출력한다 (실 운영에서 발생 빈도 모니터링).
- **uptime 정확도**: 카테고리 변경시각의 uptime 도 polling 시작 기준이 아닌 `openDate`
  기준으로 계산되므로, 단절 후 재개되어도 실제 방송 경과 시간이 정확하게 기록된다.
- 방송시작시간은 가능하면 치지직 `openDate` 를, 없으면 KST 폴백을 사용.

### 자동 기록 흐름
> 방송감지 On 시: LIVE 감지 → 행 추가(방송일/방송시작시간/영상제목) → 카테고리 변경마다 타임라인 한 줄 누적
> **카테고리 변경 즉시 시트에 자동 patch** (로컬 누적 후 일괄 업로드가 아님) — `t12` 펜딩
> 다시보기는 2행 구조가 아닌 **1행 = 1방송 세션** (담당자 분리 없음)

## 5) 컬럼 편집 규칙 (Notion형)

- 컬럼 추가/삭제/이름 변경 가능
- 컬럼 생성 시 타입을 반드시 지정
- 타입 변경 시 마이그레이션 규칙 필요
  - `text -> select` 전환 시 값 매핑 확인
  - `date -> text` 전환은 허용하되 포맷 경고
- 시스템 컬럼(예: 최종수정시각/최종수정자)은 삭제 금지 옵션 권장

## 6) Google Sheets 연동 정책

### 6-1. 설정
- 설정에서 시트 링크 저장
- 링크 유효성 검사(문서 ID 파싱)

### 6-2. 동작
- 가져오기: 시트 값을 앱 로컬 상태로 반영
- 내보내기: 앱 값을 시트로 반영
- 충돌 처리:
  - 1차 배포: 단일 사용자(관리자) 가정 → 충돌 거의 없음
  - **2차 배포 시점에 반드시 해결 필요**: 다중 사용자(편집자) 도입 시 동시 수정 → 데이터 덮어쓰기 위험 존재
  - 해결방안: 부분 업데이트 프로토콜(`6-6`) + `updatedAt` 기준 충돌 검출
  - 추후: 행 단위 충돌 UI 추가

### 6-3. 앱 시작 시 자동 동기화

앱을 처음 실행하면 **자동으로 시트에서 데이터를 내려받는다**.
수동 가져오기/내보내기도 별도로 존재.

### 6-4. 동기화 상태 표시 (UI)

| 상태 | 표시 텍스트 | 설명 |
|------|-----------|------|
| 내려받는 중 | `시트 내려받는 중...` | 시트 → 앱 데이터 가져오기 진행 중 |
| 올리는 중 | `시트 올리는 중...` | 앱 → 시트 데이터 내보내기 진행 중 |
| 완료 | `동기화 완료` | 가져오기/내보내기 성공 |
| 오류 | `동기화 실패` | 오류 발생 시 |

- 동기화 진행 중에는 **진행률(%)** 표시
  - 관리자 앱: 탭(시트) 3개 기준(숏폼/롱폼/다시보기)으로 계산
  - 편집자 앱: 탭(시트) 2개 기준(숏폼/롱폼)으로 계산
- 마지막 동기화 시각도 함께 표시

### 6-5. 진행률 계산 규칙 (확정)

#### 관리자 앱 (3탭)

- 진행 단위: `숏폼_YYYY`, `롱폼_YYYY`, `다시보기_YYYY`
- 계산식: `(완료된 탭 수 / 3) * 100`
- 예시:
  - 숏폼만 완료: `33%`
  - 숏폼 + 롱폼 완료: `66%`
  - 3개 탭 완료: `100%`

#### 편집자 앱 (2탭)

- 진행 단위: `숏폼_YYYY`, `롱폼_YYYY` (다시보기 제외)
- 계산식: `(완료된 탭 수 / 2) * 100`
- 예시:
  - 숏폼만 완료: `50%`
  - 숏폼 + 롱폼 완료: `100%`

#### 공통 UI 표기

- 내려받기 중: `시트 내려받는 중... 50%`
- 올리기 중: `시트 올리는 중... 50%`
- 모든 대상 탭 완료 시: `동기화 완료`
- 실패 시: `동기화 실패 (실패 탭명/사유)`

### 6-6. 부분 업데이트(편집자 업로드) 프로토콜

편집자 앱은 `시작일`, `마감일`, `진행상태`, `납품상태` 4개 열만 업로드한다.

#### 왜 고유키가 필요한가

편집자 앱이 부분 열만 업데이트하려면, 시트의 정확한 행을 식별해야 한다.
단순히 `방송날짜 + 영상제목`만으로는 동명이슈/날짜중복으로 충돌 가능성이 있다.

#### 권장 행 식별 키 (시트 시스템 컬럼)

숏폼/롱폼 시트에 아래 시스템 컬럼을 추가한다(일반 사용자 UI에서는 숨김 권장):

| 컬럼 | 용도 | 예시 |
|------|------|------|
| `taskId` | 영상 단위 고유 ID (2행 공유) | `tsk_20260408_ab12` |
| `roleType` | 담당 역할 식별 | `thumbnailer` / `editor` |
| `rowId` | 행 단위 고유 ID | `row_7f3k` |
| `updatedAt` | 마지막 수정시각 | `2026-04-08T11:22:33+09:00` |
| `updatedBy` | 수정 주체 | `admin:inel` / `editor:park` |

#### 편집자 업로드 절차

1. 앱은 로컬 변경행에서 `rowId` 또는 `(taskId + roleType)`를 키로 추출
2. 시트에서 동일 키 행을 조회
3. 대상 행의 4개 열만 patch 업데이트
4. 성공 시 `updatedAt`, `updatedBy` 갱신
5. 충돌/미존재 시 사용자에게 행 단위 오류 표시

#### 충돌 정책

- 기본 정책: 마지막 저장 우선(last write wins)
- 단, 키 미일치/행 미존재는 덮어쓰기 금지 후 오류 처리
- 오류 난 행만 재시도 가능하게 큐로 보관

## 9) 배포/설치 방식

- 목표 산출물:
  - Windows 설치 파일(Installer): `release/Inel Work Scheduler-Setup-<version>.exe`
  - 설치 후 exe 실행
- 권장 빌드:
  - Electron Builder 기반 `nsis` 설치 패키지 (`npm run dist`)
- asar 암호화:
  - `asarmor` 적용하여 리버스 엔지니어링 방지 (2차 배포 시점)
  - 내장된 Service Account JSON 등 민감 데이터 보호

### 9-1. 인스톨러 사양 (1차 배포 / 관리자용)

- 빌드 명령: `npm run dist` (= `vite build` + `electron-builder --win`)
- NSIS 옵션 (`package.json` > `build.nsis`):
  - `oneClick: false` — 다단계 설치 마법사
  - `perMachine: false` — 사용자별 설치 (HKCU 사용, UAC 요구 안 함)
  - `allowToChangeInstallationDirectory: true` — 설치 위치 변경 가능
  - `createDesktopShortcut: false` — 우리 커스텀 페이지에서 직접 처리
  - `runAfterFinish: true` — 설치 완료 시 실행
- 커스텀 NSIS 스크립트 (`build/installer.nsh`):
  - `preInit`: 기본 설치 경로를 **`%USERPROFILE%\Documents\Inel Work Scheduler`** 로 강제
  - `customPageAfterChangeDir`: 디렉토리 선택 다음에 옵션 페이지 추가
    - [✓] 바탕화면 바로가기 만들기 (기본 체크)
    - [✓] 윈도우 시작 시 자동 실행 (기본 체크)
  - `customInstall`: 위 체크값에 따라
    - `$DESKTOP\<productName>.lnk` 생성
    - `HKCU\Software\Microsoft\Windows\CurrentVersion\Run` 에 실행 파일 등록
  - `customUnInstall`: 두 항목 모두 제거
- 자동 실행은 앱 내 [기타 설정] 탭에서도 ON/OFF 가능
  (`app.setLoginItemSettings` IPC 사용)

### 9-2. 빌드 환경 요구사항

- 처음 빌드 시 electron-builder가 `winCodeSign` 캐시(약 5.6 MB)를 받아 풀어야 함.
  이 캐시 안에 macOS dylib **symlink** 가 포함되어 있어 Windows에서 추출 시
  Windows의 심볼릭 링크 생성 권한이 필요함.
- 다음 중 하나 충족 필요:
  1) PowerShell을 **관리자 권한으로 실행** 후 `npm run dist`
  2) Windows 11 [설정 > 시스템 > 개발자용]에서 **개발자 모드** 활성화
  - 한 번 캐시가 풀리면 그 다음부터는 일반 권한으로도 빌드 가능

## 10) 편집자용 앱 계획

### 10-1. 개요

관리자(이늘)가 편집자별 설치파일을 생성하여 배포한다.
편집자는 설치 후 실행만 하면 되며, 별도 설정이 필요 없다.

### 10-2. 관리자 앱 vs 편집자 앱

| 구분 | 관리자 앱 | 편집자 앱 |
|------|----------|----------|
| 탭 | 숏폼 / 롱폼 / 다시보기 | 숏폼 / 롱폼만 |
| 테이블 구조 | 2행 (썸네일러 + 영상편집자) | 자기 역할 행만 = 1행 |
| 업로드 상태 | 표시 + 변경 가능 | 표시만 (읽기 전용) |
| 방송감지 | O | X |
| 설정 패널 | 전체 (시트, 치지직, 감지 등) | 없음 (내장) |
| 가져오기/내보내기 | O | O (자기 담당 행 편집 → 시트 업로드) |
| 앱 시작 시 자동 동기화 | O (진행률 표시) | O (진행률 표시) |
| 편집자 관리 | O (추가/삭제/설치파일 생성) | X |
| 행/열 추가·삭제 버튼 | O | X (안 보임) |
| 시트 직접 수정 반영 | 가져오기로 반영 | 가져오기로 반영 |

### 10-2-1. 편집자 앱 권한 상세

#### 수정 가능한 열 (자기 역할 행만)

| 열 | 편집자 권한 |
|----|-----------|
| 시작일 | 수정 가능 |
| 마감일 | 수정 가능 |
| 진행상태 | 수정 가능 |
| 납품상태 | 수정 가능 |

#### 읽기 전용 (수정 불가)

| 열 | 편집자 표시 방식 |
|----|----------------|
| 업로드 | 표시만 |
| 방송날짜 | 표시만 |
| 영상제목 | 표시만 |
| 영상 카테고리 | 표시만 |
| 담당자 | 표시만 |
| 공유링크 | 하이퍼링크로 표시, 수정 아이콘 없음 |
| 납품링크 | 하이퍼링크로 표시, 수정 아이콘 없음 |

#### UI 제한

- 행 추가/삭제 버튼: **안 보임**
- 열 추가/삭제 버튼: **안 보임**
- 링크 열: 하이퍼링크 텍스트로만 표시 (클릭하면 브라우저 열림), 수정 아이콘 없음

#### 동기화 방향 제한

| 동기화 방향 | 범위 |
|-----------|------|
| 내려받기 (시트 → 앱) | **모든 행의 모든 열** 전체 내려받기 |
| 올리기 (앱 → 시트) | **수정 가능한 열만** (시작일, 마감일, 진행상태, 납품상태) |

> 관리자 앱은 **모든 열/행**에 대해 읽기/쓰기 가능, 제한 없음.
> `업로드(완)` 상태값은 관리자만 수정 가능하며, 편집자/썸네일러 앱에서는 항상 읽기 전용이다.

### 10-3. 인증 및 보안

1. **Service Account JSON 내장**: 편집자 앱 빌드 시 관리자의 Service Account JSON을 내장
2. **토큰 기반 접근 제어**: 시트 내 `_tokens` 시트에 편집자 이름 + 토큰 관리
3. **앱 실행 시 검증**: 편집자 앱은 실행할 때마다 `_tokens` 시트에서 자기 토큰 유효성 확인, 없으면 앱 잠금
4. **asar 암호화**: `asarmor` 적용으로 앱 내부 코드/데이터 리버스 엔지니어링 방지

### 10-4. 편집자 생명주기

**추가 시:**
1. 관리자 앱에서 편집자 이름 입력 → "설치파일 생성" 클릭
2. 앱이 자동으로 토큰 생성 → `_tokens` 시트에 등록 → 설치파일(.exe) 빌드
3. 설치파일 안에 시트 URL + Service Account JSON + 편집자 이름 + 토큰이 내장됨
4. 편집자에게 설치파일 전달

**사용 시:**
1. 편집자가 설치 → 실행
2. 앱이 `_tokens` 시트에서 토큰 확인 → 유효하면 숏폼/롱폼 테이블 표시
3. 상태/내용 수정 → 저장 → 시트에 업로드
4. 관리자가 시트에서 수정한 내용도 가져오기로 반영 가능

**퇴사 시:**
1. 관리자 앱에서 편집자 목록 → 해당 편집자 "삭제" 클릭
2. `_tokens` 시트에서 해당 토큰 제거
3. 편집자 앱 다음 실행 시 토큰 검증 실패 → "접근 권한이 없습니다" 표시 → 앱 잠금

### 10-5. 데이터 흐름

```
관리자 앱 ──Service Account JSON──→ Google Sheets ←── 시트 직접 편집
     ↕ 가져오기/내보내기                ↑
                              토큰 검증 후 읽기/쓰기
                                    ↑
편집자 앱 ──Service Account JSON + 토큰──→ Google Sheets
     ↕ 가져오기/내보내기(숏폼,롱폼만)
```

### 10-6. 양방향 동기화

- **편집자 → 시트**: 편집자가 앱에서 상태/내용 수정 후 저장 → 시트에 반영
- **시트 → 관리자**: 관리자가 앱에서 가져오기 → 편집자가 올린 변경사항 반영
- **시트 → 편집자**: 관리자가 시트에서 직접 수정 → 편집자가 앱에서 가져오기 → 반영

## 11) Google Sheets 시트 이름 규칙

| 탭 | 시트 이름 |
|----|----------|
| 숏폼 | `숏폼_2026` |
| 롱폼 | `롱폼_2026` |
| 다시보기 | `다시보기_2026` |

- 년도가 바뀌면 새 시트가 자동 생성됨 (예: `숏폼_2027`)
- 각 시트는 1행 = 헤더, 2행~ = 데이터
- 앱에서는 월별로 나눠 보여주지만, 시트에서는 1년치 전체가 행으로 이어짐

### 11-1. 시트 컬럼 레이아웃 (숏폼/롱폼)

권장 순서:

1. `업로드`
2. `방송날짜`
3. `영상제목`
4. `영상 카테고리`
5. `담당자`
6. `시작일`
7. `마감일`
8. `진행상태`
9. `납품상태`
10. `공유링크`
11. `납품링크`
12. `taskId` (시스템)
13. `roleType` (시스템)
14. `rowId` (시스템)
15. `updatedAt` (시스템)
16. `updatedBy` (시스템)

### 11-2. 2행 구조 검증/복구 규칙

시트 수동 편집으로 2행 구조가 깨질 수 있으므로 import 시 검증한다.

#### 정상 구조

- 홀수번째 그룹 시작행(1행째): `방송날짜` 값 존재
- 다음행(2행째): `방송날짜/영상제목/영상 카테고리/업로드` 비어있음
- `taskId` 동일, `roleType`은 각각 `thumbnailer`, `editor`

#### 비정상 사례와 처리

| 비정상 사례 | 처리 정책 |
|------------|----------|
| 2행째에 방송날짜가 입력됨 | 앱에서 경고 배지 표시 + 자동 정규화 제안 |
| 2행 쌍 중 1행이 누락됨 | 해당 그룹 오류로 분리, 수정 전까지 읽기 전용 처리 |
| `taskId` 누락 | import 시 임시 ID 부여 후 관리자에게 수정 권고 |
| `roleType` 누락/중복 | `thumbnailer/editor` 자동 추론 시도, 실패 시 오류 |

#### 정규화(옵션)

- 관리자 앱에서 `구조 정리` 실행 시:
  - 2행째 공통열 값 비우기
  - 누락된 시스템 컬럼 채우기
  - `taskId/rowId` 재생성(필요 시)

2. **구조 정리 기능**
   - 관리자 원클릭 정규화
3. **행 단위 재시도 큐**
   - 실패한 patch만 재업로드
4. **변경 이력 고도화**
   - `updatedBy`, `updatedAt` 감사로그 표기

### 테스트 기준 (수용 조건)

아래 조건을 만족하면 본 정책이 정상 반영된 것으로 본다.

1. 편집자 앱에서 시작일/마감일/진행상태/납품상태 외 셀은 편집 진입이 불가해야 한다.
2. 편집자 앱에서 공유링크/납품링크는 클릭 가능한 링크로만 보이고, 수정 아이콘이 없어야 한다.
3. 편집자 앱에서 행/열 추가삭제 UI가 나타나지 않아야 한다.
4. 편집자 앱의 업로드 요청 payload에는 4개 열만 포함되어야 한다.
5. 관리자 앱에서 업로드(완) 상태를 수정하면 편집자 앱에서 읽기 전용으로 반영되어야 한다.
6. 관리자 진행률은 3단계(33/66/100), 편집자 진행률은 2단계(50/100)로 동작해야 한다.

## 14) 편집자/썸네일러 등록 관리

### 14-1. 등록 방식

- 관리자 앱 설정 패널에서 편집자/썸네일러 목록을 관리한다.
- 직접 입력이 아닌 **등록 후 선택 방식**으로 설계하여 오타 매칭 실패를 원천 차단한다.

### 14-2. 등록 항목

| 필드 | 타입 | 설명 |
|------|------|------|
| 이름/닉네임 | text | 매칭 키. 시트의 `담당자` 컬럼과 일치해야 함 |
| 역할 | select | `썸네일러` / `영상편집자` |
| (2차) 토큰 | 자동 생성 | install 파일 빌드 시 자동 부여 |

### 14-3. 시트 `담당자` 컬럼 동작

- 컬럼 키: `assignee` (헤더 라벨: "담당자")
- UI 동작: 셀 클릭 시 등록된 인원이 **역할별로 그룹화된 드롭다운**으로 표시
- **1차 배포** (1행 = 1영상 = 담당자 1명):
  - 모든 등록 인원(썸네일러 + 영상편집자) 후보 표시
  - 그룹 헤더로 `썸네일러` / `영상편집자` 라벨 노출
- **2차 배포** (2행 구조 도입 후):
  - 1행째(썸네일러 행): 역할=썸네일러만 노출
  - 2행째(영상편집자 행): 역할=영상편집자만 노출
- 빈 값 허용 (선택 해제 버튼 제공)

### 14-4. 등록 정보 변경 시

- 이름 오타 수정: 기존 시트 데이터에 영향 → 일괄 치환 또는 시트 직접 수정 안내
- 2차 배포에서 install 파일 자동 재발급

### 14-5. 매칭 키 정규화 (1차 배포에서도 적용)

- 비교 시 양쪽 모두 `trim()` + `NFC` 정규화
- 대소문자 구분 없이 비교

## 17) AI 보조 CSV 임포트 (Monday → 우리 시트 형식 자동 변환)

### 17-1. 목적

관리자는 기존에 Monday.com 같은 외부 협업 툴을 쓰고 있었고, 그 데이터를 CSV로 내려받아 우리 형식으로 옮겨야 한다. 헤더와 값 형태가 모두 다르기 때문에 단순 매핑으로는 불가. AI에게 "헤더 ↔ 우리 컬럼" 매핑 + 값 정규화 규칙을 한 번 추출시키고, 그 규칙을 우리 코드가 모든 행에 적용한다.

### 17-2. 사용자 흐름

1. 설정 → **AI 연결** 탭에서 provider/모델/API 키 등록
2. 상단 바 또는 탭에서 **[CSV 가져오기 (AI)]** 버튼 클릭 → 모달 오픈
3. CSV 드래그&드롭 → 헤더 + 첫 5행 미리보기
4. **[AI로 분석]** 버튼 → AI 호출 → 매핑 결과(JSON) 수신
5. 우리 코드가 모든 행에 매핑 적용 → 변환된 행 미리보기
6. **[시트에 업로드]** 버튼 활성화 (분석 성공 시) → 활성 탭에 행 추가 + Sheets export

### 17-3. AI 연결 설정

| 항목 | 형태 | 비고 |
|---|---|---|
| Provider | dropdown | `OpenAI` / `Anthropic (Claude)` / `Google (Gemini)` |
| API Key | password input | localStorage 저장 (Phase 2: Electron `safeStorage` 암호화 이전 예정) |
| Model | dropdown | provider별 최신 5개 자동 갱신. 기본은 가장 최신 |
| 모델 갱신 | button | provider의 models API를 호출해 최신 5개를 다시 가져옴. 가장 최신 + 그 직전 버전들이 정렬되어 노출 |
| 가이드 | link | 외부 HTML 가이드 (`public/help/ai-setup.html`) — provider별 키 발급법 + 비용 안내 |

### 17-4. 모델 목록 fetch

| Provider | Endpoint | 인증 |
|---|---|---|
| OpenAI | `GET https://api.openai.com/v1/models` | `Authorization: Bearer KEY` |
| Anthropic | `GET https://api.anthropic.com/v1/models` | `x-api-key: KEY`, `anthropic-version: 2023-06-01` |
| Google | `GET https://generativelanguage.googleapis.com/v1beta/models?key=KEY` | URL 쿼리 |

- 각 응답에서 `created` 또는 `version` 기준 내림차순 정렬 → 상위 5개를 dropdown에 표시
- chat / generateContent를 지원하는 모델만 필터 (TTS/embedding 등 제외)
- 로컬 캐시: `userData/ai-model-cache.json` (provider별 + 마지막 갱신 시각)

### 17-5. AI 분석 요청 (CSV → 매핑 JSON)

**프롬프트(요지)**:
- 입력: CSV 헤더 + 샘플 5~10행 + 우리 표준 컬럼 스키마
- 출력 JSON 스키마:
  ```jsonc
  {
    "headerMap":   { "<csvHeader>": "<ourColumnKey or null>" },
    "valueMaps":   { "<ourColumnKey>": { "<csvValue>": "<ourValue>" } },
    "dateFormat":  { "<ourColumnKey>": "YYYY-MM-DD | MM/DD/YYYY | ..." },
    "splitColumns":{ "<ourColumnKey>": { "delimiter": ",", "trim": true } },
    "ignoreColumns": [ "<csvHeader>", ... ],
    "twoRowAssignment": { "thumbnailerColumn": "<csvHeader|null>", "editorColumn": "<csvHeader|null>" },
    "notes": "AI가 발견한 주의사항"
  }
  ```
- provider별 JSON 강제 모드:
  - OpenAI: `response_format: { type: "json_object" }` (chat) / `text.format.type: "json_object"` (responses)
  - Anthropic: 시스템 프롬프트로 `JSON only` + `tools` 없이
  - Google: `generationConfig.responseMimeType: "application/json"`

### 17-6. 매핑 적용 (우리 코드)

받은 JSON을 코드가 적용:

1. **headerMap**으로 csv 컬럼 → 우리 컬럼 키 매칭
2. **valueMaps**으로 enum 변환 (예: `"대기"→"Wait"`)
3. **dateFormat**으로 날짜 정규화 (YYYY-MM-DD)
4. **splitColumns**으로 multi-tag 분리 (`videoCategory`)
5. **twoRowAssignment**으로 썸네일러/영상편집자 행 생성
6. 매핑 안 된 컬럼은 ignore + 디버그 로그에 기록

### 17-7. 에러 / 디버그

- 분석 실패 시:
  - 모달에 "AI 분석 실패" 빨간 배지
  - **시트에 업로드 버튼 비활성화**
  - 디버그 패널에 단계별 상세 로그 출력
- 디버그 패널 우상단 **[전체 복사]** 버튼으로 클립보드 복사 → 사용자가 개발자에게 즉시 전달 가능
- 로그 카테고리: `[AI:list-models]`, `[AI:analyze]`, `[AI:apply]`, `[AI:upload]`

### 17-8. 보안 / 프라이버시

- API 키: 관리자 본인 키 (BYOK). 우리 서버 경유 없음.
- 전송 데이터: **헤더 + 샘플 5~10행만** AI에 전송 (전체 행 X). 매핑 추출 후 모든 행은 로컬에서 변환.
- Phase 2 예정: Electron `safeStorage`로 키 암호화 저장.

### 17-9. 작업 분리

| 단계 | 내용 |
|---|---|
| 1 | 설정 [AI 연결] 탭 + 외부 가이드 페이지 + 모델 갱신 IPC |
| 2 | CSV 임포트 모달 + AI 분석 IPC + 매핑 적용 |
| 3 | 디버그 로그 [전체 복사] + 카테고리화 |
| 4 (선택) | safeStorage 키 암호화 + 분석 결과 "프로필" 저장/재사용 |
